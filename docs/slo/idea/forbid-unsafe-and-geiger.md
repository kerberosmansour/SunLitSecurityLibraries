# Idea — `#![forbid(unsafe_code)]` everywhere true + `cargo-geiger` in supply-chain

**Slug:** `forbid-unsafe-and-geiger`
**Created:** 2026-05-05
**Status:** Pre-research — feeds `/slo-research` then `/slo-plan` (no `/slo-architect` — feature addition to existing system)
**Source:** awesome-rust-security-guide §3.2 + §4 cross-walk. Minimizing unsafe is a stated Rust security practice; SunLit has not made this an explicit, machine-enforced guarantee.

---

## Wedge

SunLit is a Rust security workspace whose entire selling point is memory-safe, ownership-checked code. Yet there is no compiler-level enforcement that the code stays unsafe-free, and no observability into how much unsafe lives in the dependency graph. Concrete change:

- Every published crate that does not legitimately need `unsafe` gets `#![forbid(unsafe_code)]` at the lib-root. Clippy's `unsafe_code = "deny"` is a weaker statement (overridable per-call-site); `forbid` cannot be silently bypassed.
- The rare crate that does need unsafe (likely none, but `secure_data` with `aws-lc-rs`-FFI is the candidate to audit) switches to `#![deny(unsafe_code)]` with `#[allow(unsafe_code)]` at justified, named call sites — every site documented with the security argument and a regression test.
- `cargo-geiger` is added to `scripts/audit.sh` and the supply-chain CI lane to track transitive unsafe in dependencies. Output published as a build artifact; threshold gating is a follow-up if the metric proves stable.

This is mostly a documentation+config milestone — but it bakes "no unsafe" into the contract instead of into the developer's good intentions.

## Target user

(1) The maintainer adopting an explicit safety invariant once; (2) downstream consumers and certifiers (SOC 2, IEC 62443, FIPS) who want to cite "all crates compile-time forbid `unsafe` except the documented exceptions" as a defensible statement, with `cargo-geiger` numbers backing the dependency-side claim.

## Why this is non-trivial

- **`forbid(unsafe_code)` is total.** It cannot be locally overridden with `#[allow(unsafe_code)]`. If any crate has `unsafe` blocks in transitive `#[allow]` it will refuse to compile. The audit must be careful — the change is non-reversible without dropping back to `deny`.
- **Macro-expanded unsafe is invisible to humans.** Some macros (especially in `serde_derive`, `tokio::pin!`, `pin-project`) expand to unsafe blocks. Whether `forbid` rejects the expansion or only the literal `unsafe` keyword has rustc-version-specific behavior. Worth understanding before flipping a switch.
- **`cargo-geiger` reads transitive code.** The number it emits is sensitive to feature flags. A `cargo-geiger --all-features` run against the workspace will look very different from default features. Picking which feature set to publish as the project's "official unsafe usage" number is a design decision.
- **The supply-chain lane already does cargo-audit + cargo-deny + cargo-vet.** Adding cargo-geiger should follow the same pattern (CI-installed, scripted, output preserved as artifact); no new abstractions.

## What "done" looks like

A 2–3 milestone runbook:

1. **M1** — Audit every published crate's lib-root for `unsafe_code` use and macro-expanded unsafe via `cargo expand`. For each: decide forbid-vs-deny. Add the attribute. Add a CI test (`compile_fail` doctests or a `tests/no_unsafe_code.rs` equivalent) that catches regression. Document each `#[allow(unsafe_code)]` call site (if any) with the security argument.
2. **M2** — Add `cargo-geiger` to `scripts/audit.sh` and the `supply-chain` CI job. Pin the version. Decide and document the feature-set the official number uses. Publish geiger output as a CI artifact. Add a README badge or table row.
3. **M3 (optional)** — Add a "transitive unsafe budget" — a documented threshold above which the supply-chain CI fails. Initial threshold = current measured value + small headroom. Re-evaluate quarterly.

## Open questions for /slo-research

1. **Macro-expanded `unsafe` and rustc version interaction with `forbid(unsafe_code)`.** Specifically: as of stable Rust 1.83+ (2026 LTS-ish), does `forbid(unsafe_code)` reject `unsafe` blocks generated by procedural macros, or only literal `unsafe` in source? Drives M1's go/no-go on every crate and the choice of `forbid` vs. `deny`. Look for: rustc reference for `forbid_unsafe_code` lint, recent issues on `rust-lang/rust` about macro-expanded unsafe and the lint, blog posts from major Rust security crates that have flipped this switch.

2. **Whether SunLit's actual macro dependencies expand to `unsafe`.** Specifically: `serde_derive`, `thiserror`, `tokio::pin!`, `axum`'s extractor macros, `tracing::instrument`, `serde_json` derive paths. Drives the audit list. Method: run `cargo expand` per crate and grep for `unsafe`. Tool-driven; minimal external research, but the *answer* is project-internal. Look for: `cargo-expand` README for current invocation, `serde_derive` source to understand whether its expansion uses unsafe.

3. **Current `cargo-geiger` maintenance status and recommended replacement (if abandoned).** Per public crates.io page, `cargo-geiger` had reduced activity through 2024–2025. Drives M2: use cargo-geiger or pivot to a successor (`cargo-supply-chain`, `cargo-attribute-stats`, or a hand-rolled rustc lint counter). Look for: cargo-geiger crates.io page, recent commits, GitHub issues and discussion of forks/successors.

4. **Recommended `cargo-geiger` invocation for a multi-crate workspace.** Specifically: `--all-features`? `--no-default-features`? per-crate iteration? Output format (JSON for CI vs. human-readable for README)? Drives M2's scripts/audit.sh recipe and CI artifact format. Look for: cargo-geiger README/wiki, real-world workspace usage examples (rustls, hyper).

5. **Examples of high-profile Rust crates that ship `forbid(unsafe_code)` AND publish their cargo-geiger numbers.** Drives M3's threshold-policy framing — if the field-current best-in-class number for a similar-sized workspace is X, our threshold should be X-band. Look for: rustls (commonly cited), `tower`, `axum`, `serde` (derive subcrate vs. main), recent OWASP / RustSec blog posts naming exemplars.

## Out of scope for research

- Replacing `cargo-audit` / `cargo-deny` / `cargo-vet` — those are working and out of scope.
- Whether to enable `clippy::pedantic` or `clippy::nursery` — separate hardening exercise.
- `unsafe_op_in_unsafe_fn` lint — useful, but separate; flipping that is a different runbook.
- Compile-time vs. runtime safety analysis (loom, miri-on-unsafe) — already on roadmap or in place.

## Constraints

- Cannot break any existing crate's build with feature combinations that consumers already use.
- Documentation of `#[allow(unsafe_code)]` exceptions is mandatory — silent allow is forbidden.
- Geiger CI lane runs as advisory before becoming gating (M3).
- Threshold (M3) must be derived from actual measured baseline, not aspirational.

## Success criteria

After research and runbook:
- Every published crate has either `#![forbid(unsafe_code)]` or `#![deny(unsafe_code)]` with a documented allow-list.
- `cargo-geiger` runs in CI, output published, README states the workspace's transitive-unsafe number.
- A `tests/no_unsafe_code.rs`-equivalent regression test exists per crate (or a workspace-level equivalent) that fails the build if the attribute is removed.
- The supply-chain CI lane has a new job step; existing `cargo-audit` / `cargo-deny` / `cargo-vet` jobs are unchanged.
