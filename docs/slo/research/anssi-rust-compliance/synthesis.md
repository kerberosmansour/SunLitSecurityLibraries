# Synthesis — anssi-rust-compliance

## 1. ANSSI guide pinning strategy

The ANSSI Rust guide is a living document with no semantic versioning. The repo has one Git tag (`v1.0` at `28076d10f274`, 2020-06-15); the published rendering is labelled "Secure Rust Guidelines (unstable)". Master moves sub-monthly (latest 2026-04-07 `84e6ae181712`, 2026-03-25, 2026-03-23) and CONTRIBUTING.md calls it a "living document". The mapping must therefore pin to a commit SHA (not a tag), record SHA + commit date, and be revalidated quarterly via `git log master --since=<last-pin>`, **so the design must handle commit-pinned rebuild semantics with a recorded refresh cadence because [the ANSSI-FR/rust-guide commits API](https://api.github.com/repos/ANSSI-FR/rust-guide/commits/master) shows ongoing master churn with no tagged release process and the site is labelled "(unstable)".**

## 2. Mapping table column shape

The idea-doc proposed `Rule ID | Rule (English) | Status | Evidence | Notes`. Rules use one English-slug ID (`id="<FAMILY>-<SLUG>"`); the French rendering reuses the same slug, so no separate French ID column is needed. The `type` attribute carries three values (`Rule`, `Recommendation`, `Warning`) which a faithful mapping must preserve, since Recommendation rows carry weaker compliance pressure than Rule rows. No public Rust project ships a per-rule mapping table to copy from. **The design must handle a six-column shape `Rule ID | Rule title | Type (Rule/Recommendation/Warning) | Status (compliant/partial/waived/N-A) | Evidence | Notes` because [src/en/unsafe/memory.md @ 84e6ae18](https://raw.githubusercontent.com/ANSSI-FR/rust-guide/84e6ae181712/src/en/unsafe/memory.md) shows `type="Rule"` vs `type="Recommendation"` is an authored distinction reviewers consume.**

## 3. Cross-walk leverage to EU regulations

Procurement value depends on whether EU regulation cites the guide normatively. It does not. NIS2 Article 21 implementing acts ([Commission Implementing Regulation (EU) 2024/2690](https://eur-lex.europa.eu/eli/reg_impl/2024/2690/oj)) define "technical and methodological requirements" without naming language guides; ENISA's June 2025 Technical Implementation Guidance is generic. The CRA ([Regulation (EU) 2024/2847](https://eur-lex.europa.eu/eli/reg/2024/2847/oj)) requires Annex I conformance, but standardisation request M/606 enumerates 41 standards none of which is the ANSSI guide. Leverage is *indirect*: admissible as "state of the art" / national-CA evidence in French markets and as the documented Rust secure-coding standard inside an IEC 62443-4-1 SD-3 audit, not as a directly-cited harmonised standard. **The design must handle README/THREAT_MODEL framing precisely — the correct claim is "documented secure-coding evidence usable under NIS2 national implementation, CRA Annex I state-of-the-art demonstration, and IEC 62443-4-1 SD-3" rather than "ANSSI compliance maps to NIS2/CRA" because [the EU CRA standardisation page](https://digital-strategy.ec.europa.eu/en/policies/cra-standardisation) lists 41 standards but does not name ANSSI guidance.**

## 4. Tooling-evidence convention

About 25 of 61 rules (~40%) admit an automated check via Clippy restriction-group lints, `cargo-deny`, `cargo-audit`, `cargo-outdated`, `cargo-geiger`, `forbid(unsafe_code)`, or Semgrep `p/rust`. The remaining ~36 (mostly FFI) need code review or doc-only evidence. Pointers should be machine-resolvable: lint names verified by `cargo clippy -- -W clippy::<name>`, advisory IDs by `deny.toml`, file:line by a CI script. **The design must handle a heterogeneous evidence column with an enumerated pointer set (`lint:<name>`, `tool:<name>`, `file:<path>:<line>`, `test:<path>::<name>`, `doc:<anchor>`, `waiver:<id>`) plus a CI lint that resolves each, because [the Clippy lints book](https://doc.rust-lang.org/clippy/lints.html) and [cargo-deny docs](https://embarkstudios.github.io/cargo-deny/) only cover ~40% of the corpus and the rest will rot silently without machine-checkable pointers.**
