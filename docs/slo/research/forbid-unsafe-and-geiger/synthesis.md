# Synthesis — `forbid-unsafe-and-geiger`

## 1. `forbid` vs `deny` per crate class

The published evidence cleanly partitions Rust security/networking crates into two classes. Pure-logic and HTTP/TLS-policy crates — `rustls` (`#![forbid(unsafe_code, unused_must_use)]` at `rustls/src/lib.rs`), `axum` (README: "uses `#![forbid(unsafe_code)]` to ensure everything is implemented in 100% safe Rust"), and the entire `tower` family — all set `forbid` at their crate roots. Protocol-engine and ser/de-core crates that own pointer work — `hyper`, `serde` — do not, and rely on locally-justified `unsafe`. SunLit's published crates (`secure_errors`, `security_events`, etc.) sit in the first cluster: pure logic, no FFI other than the aws-lc-rs glue in `secure_data`. The recommendation is therefore `#![forbid(unsafe_code)]` at every crate root *except* `secure_data` if and only if it must hand-write FFI shims — `secure_data` should switch to `#![deny(unsafe_code)]` with a documented `#[allow(unsafe_code)]` per FFI call site, mirroring the hyper pattern. **The design must handle the forbid-vs-deny split per-crate because the rustc lint reference (`doc.rust-lang.org/rustc/lints/levels.html`) makes `forbid` non-overridable, and the ANSSI guide's `LANG-UNSAFE` rule explicitly mandates `forbid` at the crate root for any crate that does not require unsafe.**

## 2. `cargo-geiger` go/no-go and invocation

`cargo-geiger` is active but partial: v0.13.0 (2025-08-31) and v0.12.0 (2025-04-10) are compat-bumps for Rust 1.89 / Cargo 0.86 rather than feature work, with a v0.13.0 milestone due 2026-01-01 still 8% complete. There is no abandoned-status banner and CI runs continuously, so it is the right go-forward choice — with a documented fallback (`rg`-based `unsafe`-counter over `cargo metadata`'s resolved graph) ready for the day a rustc bump breaks it. For SunLit's workspace, the correct invocation is `cargo geiger --workspace --all-features --output-format Json` from the workspace root, captured to a CI artifact, with a parallel ASCII run for the README. `--all-features` is the conservative number — it expands every cfg-gated dep, making the metric an upper bound on transitive unsafe surface; `--no-default-features` produces a smaller, less defensible number because consumers compile with their own feature set. **The design must handle `--all-features` as the canonical setting because `cargo-geiger`'s own CLI source (`cargo-geiger/src/cli.rs`) supports it as the conservative metric, and the LogRocket supply-chain comparison treats geiger as a worst-case unsafe-surface scanner rather than a per-consumer report.**

## 3. CI integration — placement, gating, threshold

`cargo-geiger` belongs in the existing `supply-chain` CI lane next to `cargo-audit` / `cargo-deny` / `cargo-vet`, installed via a prebuilt-binary action (`taiki-e/install-action` or `baptiste0928/cargo-install`) to avoid the multi-minute compile cost. It must run **advisory first**, converting to blocking only after the threshold is calibrated against an actual measured baseline. Reference projects do not give a number to copy: rustls's `build.yml` runs cargo-audit, miri, fuzz, semver-checks, and external-types but does *not* run cargo-geiger; axum relies entirely on compile-time `forbid(unsafe_code)`. So the SunLit threshold must be derived from the first measured workspace value plus documented headroom (per idea-doc M3 constraint). The CI step's JSON should be uploaded as an artifact for every run, and the README badge / line should track the number publicly so downstream auditors can cite it for IEC 62443-4-1 secure-coding evidence and as supporting material in a FedRAMP SC-39/SI-16 control narrative. **The design must handle the advisory-then-blocking ramp because rustls's published CI demonstrates that mature security crates do not gate on geiger numbers (only on compile-time forbid), and the LogRocket supply-chain comparison shows the geiger metric is best treated as auditor input rather than a hard gate.**

## 4. Macro-expansion risk surface

Among SunLit's macro-deps, two are load-bearing for the audit. `tokio::pin!` *does* emit literal `unsafe { Pin::new_unchecked(…) }` in the consumer crate's expansion (verified at `docs.rs/tokio/latest/src/tokio/macros/pin.rs.html`) and will break `forbid(unsafe_code)` if invoked from a forbid-set crate; the runbook must either ban `tokio::pin!` from forbid-set crates or substitute `pin-project-lite`, which empirically works under `forbid` in axum/tower because its `unsafe` is hygienic to its own crate. `serde_derive`, `thiserror`, `tracing::instrument`, and `axum-macros` derives all emit safe-only expansions (verified via `serde_derive/src/lib.rs` and `thiserror/impl/src/expand.rs`) and pose no risk. The remaining subtle case — `#[allow_internal_unsafe]`-flagged std macros (`thread_local!`, historical `await!`) — evades `forbid` by design (rust-lang/rust#56768) but does not affect SunLit because SunLit does not author such macros. **The design must handle `tokio::pin!` as a forbid-breaker because the macro's source explicitly expands to `unsafe { Pin::new_unchecked(&mut $x) }`, and rust-lang/rust#56768 documents that `forbid(unsafe_code)` does not catch the small set of `allow_internal_unsafe`-flagged macros that bypass the lint.**
