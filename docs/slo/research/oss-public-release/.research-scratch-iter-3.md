## Deepened Findings

### Repo grounding (high confidence — sourced from the working tree)

- All 13 crates use `edition = "2021"` and `license = "MIT"` (single, not dual). No `rust-version` set anywhere. M1 must add LICENSE-MIT + LICENSE-APACHE, flip every manifest to `MIT OR Apache-2.0`, and add `package.rust-version`.
- **8 of 13 crates already have in-tree `cargo-fuzz` harnesses (18 fuzz targets total)**: `secure_boundary` (4), `secure_data` (2), `secure_identity` (1), `secure_network` (3), `secure_output` (2), `secure_privacy` (2), `secure_resilience` (1), `security_events` (2). This materially shrinks M4 — the gap is `secure_authz` coverage and a CI nightly, not greenfield infra.
- Only **2 occurrences of `unsafe`** across the workspace — geiger badge will read very low; do not over-promise.
- `secure_data` uses `aws-lc-rs` (FIPS feature), `secure_identity` pulls `ring 0.17` — both bring C/CMake toolchains, decisive for the reproducibility question.
- CI actions in `.github/workflows/ci.yml` are unpinned (`@v4`, `@v2`, `@stable`) — SHA pinning + Dependabot for `github-actions` is real M2 work.
- DAST workflows already exist; M5 needs a gating policy, not creation.

### Q1 — MSRV floor (medium confidence; project-specific)

`cargo-msrv` **0.19.3** (2026-03-25) is current ([foresterre/cargo-msrv](https://github.com/foresterre/cargo-msrv)). No 2026 community-blessed numeric default exists. Two reference policies: **RustCrypto** (MSRV bumps are breaking, tested in CI; [PR #351](https://github.com/RustCrypto/AEADs/pull/351)) and **latest-stable − N** (Tokio-style). Recommendation: pick floor via `cargo msrv find` once, declare in every crate's `package.rust-version`, gate CI with `cargo msrv verify`, document a RustCrypto-style "MSRV is breaking" policy. RFC 3537 (MSRV-aware resolver) is now in Cargo and reduces the cost of a conservative floor ([RFC 3537](https://rust-lang.github.io/rfcs/3537-msrv-resolver.html)).

### Q2 — OpenSSF Scorecard ceiling for solo maintainers (high confidence on mechanics)

**Code-Review uses tiered deductions, not proportional scoring**: −7 points for one unreviewed human change, −3 more for multiple, −3 for unreviewed bot changes ([Scorecard checks.md](https://github.com/ossf/scorecard/blob/main/docs/checks.md)). For a solo maintainer who self-merges, the structural floor on Code-Review is **0/10** — CODEOWNERS doesn't satisfy it; an actual approving review by a different account does. **Contributors** requires "≥ 3 different companies in last 30 commits, each with ≥ 5 commits" for max. Inference: solo Rust security repo executing every other check well realistically tops at **6–7/10 overall**. **Revise iter-2's 8/10 stretch target down to 7/10** unless a co-maintainer joins. Don't advertise a target score in README.

### Q3 — `cargo-semver-checks` and feature-gated APIs (high confidence)

Supports `--cfg` via `RUSTDOCFLAGS`; current line is 0.46+ with ~245 lints; MSRV is Rust 1.90 ([cargo-semver-checks](https://github.com/obi1kenobi/cargo-semver-checks)). For this feature-matrix workspace, a single `--workspace` invocation misses feature-gated regressions — needs `--all-features` and `--no-default-features` legs at minimum.

### Q4 — OSS-Fuzz onboarding turnaround (low–medium confidence)

Acceptance is discretionary on a "criticality score"; new repo with zero downstream dependents is unfavorable. No 2026 SLA surfaced. **OSS-Fuzz is realistically post-1.0.** The existing in-tree directory layout (`crates/*/fuzz/Cargo.toml` + `fuzz/fuzz_targets/*.rs`) is exactly what OSS-Fuzz expects — onboarding is a `project.yaml` + `Dockerfile` + `build.sh` PR when the time comes ([OSS-Fuzz Rust guide](https://google.github.io/oss-fuzz/getting-started/new-project-guide/rust-lang/)).

### Q5 — `harden-runner` audit-mode burn-in (medium confidence)

Documented protocol is "audit, observe across multiple runs, then transition using Recommendations tab" ([StepSecurity docs](https://docs.stepsecurity.io/harden-runner)). No fixed duration upstream. For this 3-OS Rust matrix: **5–10 distinct runs across all three OS, ~1 week** is the practical floor. Allowlist must include `crates.io`, `static.crates.io`, `index.crates.io`, `github.com`, `objects.githubusercontent.com`, plus `Swatinem/rust-cache` endpoints.

### Q6 — Reproducible-build feasibility (high confidence: NOT cross-platform feasible in 2026)

Rust toolchain doesn't support reproducible builds out-of-the-box ([rust-lang/rust#34902](https://github.com/rust-lang/rust/issues/34902)). March 2026 cargo PR ([#16691](https://github.com/rust-lang/cargo/pull/16691)) is in flight but not merged. With `aws-lc-rs` and `ring 0.17` pulling C/CMake toolchains, cross-platform Ubuntu/macOS/Windows reproducibility is unachievable. **Achievable bar: single-platform single-runner reproducibility** (pinned Ubuntu container, pinned toolchain) — sha256-compared between two consecutive runs.

### Q7 — Trusted Publishing first-publish gotcha (high confidence — newly surfaced)

**New crates cannot be first-published via Trusted Publishing**; first version must use a long-lived token, then TP takes over for subsequent versions ([crates.io TP docs](https://crates.io/docs/trusted-publishing); [release-plz quickstart](https://release-plz.dev/docs/github/quickstart)). Iter-2's "no interim long-lived-token posture" claim was wrong for first launch. Updated launch sequence: (1) one-shot manual bootstrap publish for all 13 crates, (2) register each crate's TP relationship, (3) enable per-crate enforcement mode ([crates.io dev update 2026-01-21](https://blog.rust-lang.org/2026/01/21/crates-io-development-update/)), (4) delete the bootstrap token.

### Reaffirmed from iter-2

License default `MIT OR Apache-2.0`; SLSA L2 standalone vs L3 via reusable workflow; CodeQL Rust GA in 2.23.3; Semgrep Rust GA; OSV-Scanner v2.3.5 covers Cargo; harden-runner is cross-platform; all four cargo-vet `[imports]` feeds live; Cosign 2.4.x is current with bundle-format verify.

## Library Evaluations

Iter-2 evaluations stand. Iter-3 deltas:

**`cargo-msrv` 0.19.3** — `cargo msrv verify` is the drop-in CI gate; needs an MSRV chosen first. **Use** in dedicated CI job after `cargo msrv find` discovery.

**`cargo-semver-checks` 0.46+** — runner needs Rust ≥ 1.90; single `--workspace` invocation misses feature-gated regressions. **Use** post-1.0 only as a release-gate, not PR-gate; matrix over `--all-features` + `--no-default-features`.

**`release-plz`** — native TP support; **first publish per crate must be manual**; this is a crates.io limitation. **Use** for steady state after bootstrap; pair with crates.io's per-crate enforcement toggle.

**Cosign 2.4.x** — 2.4.0 added bundle-format verify for GitHub Artifact Attestations. **Use**; document the exact `verify-blob` invocation including OIDC issuer and identity regex in README.

**Reproducible-build tooling (diffoscope / sha256 compare)** — feasible single-platform only on pinned Ubuntu; cross-OS infeasible due to `aws-lc-rs`/`ring`. **Use** scoped to one canonical platform; defer multi-platform to post-1.0 once cargo PR #16691 stabilises.
