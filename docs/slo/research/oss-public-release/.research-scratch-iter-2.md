## Deepened Findings

**License default — MIT OR Apache-2.0 confirmed for 2026 (high confidence).** No evidence of community drift. Apache-2.0 supplies the explicit patent grant; MIT preserves GPL-2.0/LGPL-2.1 compatibility (Apache-2.0 alone is GPL-2-incompatible). The Rust API Guidelines and Google's third-party Rust guidance still call for the dual default ([Rust API Guidelines — Necessities](https://rust-lang.github.io/api-guidelines/necessities.html); [Google Rust OSS guidance](https://opensource.google/documentation/reference/thirdparty/rust); [Rust Internals — Rationale of Apache dual licensing](https://internals.rust-lang.org/t/rationale-of-apache-dual-licensing/8952)). All 13 crates currently declare `license = "MIT"` (per readiness §2) — the M1 fix is a manifest-wide replace plus two LICENSE files.

**`actions/dependency-review-action` v4.9.0 (2026-03-03) supports Cargo via GHSA (high confidence).** GHSA imports RustSec advisories under CC-BY-4.0, so Cargo.lock-introduced vulnerable crates are flagged. License-allowlist + `fail-on-severity` are first-class config ([repo](https://github.com/actions/dependency-review-action); third-party confirmation [Pass-the-SALT 2023 talk](https://archives.pass-the-salt.org/Pass%20the%20SALT/2023/slides/PTS2023-Talk-11-rust-supply-chain-security.pdf)). Inference (medium confidence): GHSA *can* surface advisories that have not yet landed in RustSec, so this action is genuinely additive on top of `cargo audit`.

**`step-security/harden-runner` is cross-platform (high confidence).** macOS + Windows runner support is now GA, matching the Linux experience with no syntax change. Critical for this repo: the existing CI matrix runs `ubuntu-latest`, `macos-latest`, `windows-latest`, so harden-runner can cover every job ([StepSecurity blog — Windows + macOS support](https://www.stepsecurity.io/blog/harden-runner-now-supports-windows-and-macos-github-actions-runners); [Harden Runner repo](https://github.com/step-security/harden-runner)).

**OSV-Scanner V2 supports Rust/Cargo (high confidence).** Latest is **v2.3.5 (March 2026)** with cargo lockfile support listed alongside npm/pip/maven/etc. Provides genuinely complementary coverage to `cargo audit` because it queries OSV.dev, which aggregates GHSA + RustSec + others ([OSV-Scanner V2 announcement](https://security.googleblog.com/2025/03/announcing-osv-scanner-v2-vulnerability.html); [google/osv-scanner](https://github.com/google/osv-scanner); [docs](https://google.github.io/osv-scanner/)).

**CodeQL Rust is GA, Semgrep Rust is GA — both viable defaults (high confidence).** CodeQL Rust GA in 2.23.3 (2025-10-23). Inference: native Security-tab integration tilts the choice toward CodeQL as the *primary*; running Semgrep alongside is the LinkedIn pattern and adds inexpensive coverage for OWASP-style rules ([CodeQL 2.23.3 changelog](https://github.blog/changelog/2025-10-23-codeql-2-23-3-adds-a-new-rust-query-rust-support-and-easier-c-c-scanning/); [Semgrep Rust GA](https://semgrep.dev/products/product-updates/rust-ga-support-and-swift-beta-support/); [InfoQ — LinkedIn SAST](https://www.infoq.com/news/2026/02/linkedin-redesigns-sast-pipeline/)). For this repo: CodeQL must be added as a workflow; the existing security-related rule surface for Rust is starter-level (`rust/insecure-cookie` + ~core CWE coverage), so Semgrep `p/rust` + `p/owasp-top-10` is genuine breadth not duplication.

**SLSA Level achievable (high confidence).** `actions/attest-build-provenance@v2` *alone* meets **SLSA Build Level 2**. **Level 3 requires** the build steps to live in a *reusable workflow* (or in `slsa-framework/slsa-github-generator`) so signing material is unreachable from user-defined steps. GitHub publishes both paths; the `attest-build-provenance` + reusable-workflow combo is the documented one ([GitHub Docs — reusable workflows + attest-build-provenance for SLSA L3](https://docs.github.com/actions/security-guides/using-artifact-attestations-and-reusable-workflows-to-achieve-slsa-v1-build-level-3); [GitHub Blog — Enhance build security and reach SLSA Level 3](https://github.blog/enterprise-software/devsecops/enhance-build-security-and-reach-slsa-level-3-with-github-artifact-attestations/); [Ian Lewis explainer](https://www.ianlewis.org/en/understanding-github-artifact-attestations)).

**crates.io Trusted Publishing is GA on GitHub Actions (high confidence).** OIDC, no long-lived token; per-crate enforcement toggle; `pull_request_target` and `workflow_run` triggers are refused as TP sources; GitLab.com is public beta only ([crates.io dev update 2026-01-21](https://blog.rust-lang.org/2026/01/21/crates-io-development-update/); [RFC 3691](https://rust-lang.github.io/rfcs/3691-trusted-publishing-cratesio.html); [crates.io TP docs](https://crates.io/docs/trusted-publishing)). For this repo: mandate TP for all 13 crates; no interim long-lived-token posture is necessary.

**`cosign verify-blob` UX is unchanged (high confidence).** Still requires `--certificate-identity` + `--certificate-oidc-issuer` ([cosign verify-blob doc](https://github.com/sigstore/cosign/blob/main/doc/cosign_verify-blob.md); [Cosign 2.0 release notes](https://blog.sigstore.dev/cosign-2-0-released/)). Inference: README must document the verify command verbatim with the exact OIDC issuer (`https://token.actions.githubusercontent.com`) and identity regex pattern, otherwise downstream consumers will get useless verification or false positives. The future direction is `sigstore-go`, not a UX simplification.

**cargo-vet 0.10.2 is the current line; the repo's `version = "0.10"` declaration is correct (high confidence).** All four canonical publisher feeds (Mozilla / Google / Bytecode Alliance / Embark) are live and listed in upstream `registry.toml` ([cargo-vet 0.10.2 docs.rs](https://docs.rs/crate/cargo-vet/latest); [registry.toml](https://github.com/mozilla/cargo-vet/blob/main/registry.toml)). The existing `supply-chain/config.toml` shows ~990 exemption lines — a documented burn-down (e.g., "≤50 by 1.0") is a concrete target, not aspirational.

**OpenSSF Scorecard's "Contributors" check explicitly looks for multiple organizations, not just multiple users (high confidence).** A solo maintainer cannot score perfectly on Contributors regardless of supply-chain hygiene. Inference (medium confidence, no hard 2026 benchmark surfaced): a solo Rust security repo executing every other check well typically targets **7–8 / 10 overall**, with `Contributors` and possibly `Code-Review` (single-reviewer constraint) as the structural floors. Treat 8 as the credible "stretch" bar; don't promise 9+ ([Scorecard checks doc](https://github.com/ossf/scorecard/blob/main/docs/checks.md); [scorecard.dev](https://scorecard.dev/)).

**MSRV tooling (high confidence).** `cargo-msrv` `verify` reads `package.rust-version` and exits non-zero on failure — drop-in CI gate ([cargo-msrv book — verify](http://gribnau.dev/cargo-msrv/commands/verify.html); [foresterre/cargo-msrv](https://github.com/foresterre/cargo-msrv)). RFC 3537 (MSRV-aware resolver) is now in Cargo, which lowers the downstream cost of conservative floors. No 2026 community-blessed numeric default surfaced; pick by *toolchain features actually used* (e.g., `let-else`, `async fn` in traits).

**OSS-Fuzz acceptance is discretionary on a "criticality score" (medium confidence).** This repo's framing as "mission-critical primitives" doesn't automatically clear the bar — at announcement time the libraries have zero downstream dependents on crates.io, which is the dominant signal. Inference: in-tree `cargo-fuzz` harnesses are achievable in the time budget; OSS-Fuzz onboarding is realistically a *post-1.0* deliverable once external adoption exists ([OSS-Fuzz Rust integration guide](https://google.github.io/oss-fuzz/getting-started/new-project-guide/rust-lang/); [Trail of Bits Testing Handbook — OSS-Fuzz](https://appsec.guide/docs/fuzzing/oss-fuzz/)).

## Library Evaluations

**`actions/attest-build-provenance@v2`**
- Latest: action major `v2`; SHA-pin required.
- Pros: native Sigstore Fulcio + Rekor; integrates with GitHub's built-in attestations API; documented L3 path with reusable workflow.
- Cons: standalone tops out at SLSA L2. The L3 reusable-workflow refactor is non-trivial because every cargo step must move into a callable workflow.
- Fit: **Use.** L2 immediately; commit to L3 path before claiming "SLSA Level 3" anywhere in README.

**`slsa-framework/slsa-github-generator`**
- Status: actively maintained reusable-workflow framework.
- Pros: language-agnostic L3-out-of-the-box; well-documented.
- Cons: less Rust-specific exemplar material than the attest-build-provenance + reusable-workflow path; adds a second maintained dependency tier.
- Fit: **Backup option.** Choose only if attest-build-provenance + custom reusable workflow proves operationally heavy.

**`actions/dependency-review-action` v4.9.0**
- Pros: Cargo via GHSA mirror; license-allow/deny + severity gates first-class; PR-time block.
- Cons: Pull-request-only event; doesn't replace `cargo audit` on `main`/cron.
- Fit: **Use** as a PR gate alongside, not instead of, `cargo audit`.

**`step-security/harden-runner` (latest 2.19.x)**
- Pros: cross-platform (Linux/macOS/Windows) — covers the existing 3-OS matrix; egress audit→block transition is the documented pattern.
- Cons: audit-mode burn-in length is repo-specific; need to allowlist `crates.io`, `static.crates.io`, `index.crates.io`, `github.com`, plus any `actions/cache` endpoints.
- Fit: **Use** in `audit` mode at launch, flip to `block` after one full week of clean baselines.

**`google/osv-scanner` v2.3.5**
- Pros: aggregates OSV.dev (GHSA + RustSec + others); SARIF output; Cargo lockfile native.
- Cons: overlap with `cargo audit` and `dependency-review-action`. Three scanners can produce noisy duplicate alerts.
- Fit: **Use** but de-duplicate intentionally — e.g., `cargo audit` on cron, dependency-review on PR, OSV-Scanner weekly with SARIF upload to Security tab.

**CodeQL (Rust GA in 2.23.3)**
- Pros: native GitHub Security tab integration; high recall historically; GA.
- Cons: Rust rule library is starter-level for security-specific patterns.
- Fit: **Use** as the primary SAST.

**Semgrep (Rust GA)**
- Pros: complements CodeQL with OWASP rulesets; faster local feedback.
- Cons: separate alert surface; commercial tier exists but `p/rust` + `p/owasp-top-10` are free.
- Fit: **Use** alongside CodeQL.

**`cargo-vet` 0.10.2**
- Pros: matches existing `version = "0.10"` declaration; canonical imports live for all four orgs.
- Cons: ~990 exemptions today — burn-down is real labor.
- Fit: **Already in use; add `[imports]`.**

**`cargo-msrv` (active per March 2026 metadata)**
- Pros: `verify` exits non-zero, drop-in CI gate; understands `package.rust-version`.
- Cons: needs an MSRV picked first (project decision, not tooling).
- Fit: **Use** in dedicated MSRV CI job.

**`cargo-mutants` v27.0.0 (2026-03-07)**
- Pros: actively maintained; security-invariant tests benefit disproportionately from mutation testing.
- Cons: long runtimes; needs a curated `mutants.toml` to be CI-viable.
- Fit: **Use** scoped to security-invariant crates (`secure_boundary`, `secure_authz`, `secure_identity`, `security_events`); off the PR critical path.

**`cargo-geiger` 0.13.0 (2025-08-31)**
- Pros: simple unsafe-LOC counter still maintained; fits an "unsafe audit" report nicely.
- Cons: known to undercount expressions inside `unsafe fn` bodies; treat the number as a directional signal, not a metric.
- Fit: **Use** for periodic reporting + README badge; do not gate CI on the count.

**`actions/dependabot` (cargo + github-actions ecosystems)**
- Pros: native; required to keep SHA-pinned actions current.
- Cons: bumps trigger churn; PR review burden on a solo maintainer.
- Fit: **Use** with grouped updates and weekly cadence to limit churn.

**`taiki-e/install-action` (for SHA-pinned tool installs)**
- Pros: SHA-pinnable; supports explicit `--version`; covers `cargo-audit`, `cargo-deny`, `cargo-vet`, `cargo-llvm-cov`, `cargo-mutants`.
- Cons: another transitive trust point — must itself be SHA-pinned.
- Fit: **Use** to fix the unpinned `cargo install ...` pattern in current `supply-chain` job.

**`release-plz` vs `cargo-release`**
- Pros (`release-plz`): designed for multi-crate workspaces; auto-PR generation; first-class crates.io Trusted Publishing support.
- Cons (`release-plz`): more workflow-state to manage than `cargo-release`'s scriptable model.
- Fit: **Prefer `release-plz`** for this 13-crate workspace; `cargo-release` is the fallback for a more imperative model.

## Architecture Options

**Option A — Full "mission-critical" posture (SLSA L3 + OSS-Fuzz + dual-SAST)**
- Shape: `release.yml` is a thin caller that delegates `cargo build`/`cargo publish` to a reusable workflow housed in this repo (or `slsa-framework/slsa-github-generator`). Trusted Publishing on every crate. Cosign-keyless signs every release artifact + SBOM. Scorecard + CodeQL + Semgrep + OSV-Scanner + dependency-review + harden-runner all running. cargo-vet `[imports]` from all four orgs. In-tree `cargo-fuzz` harnesses for `secure_boundary`/`secure_identity`/`security_events`; OSS-Fuzz application opened post-launch.
- Trade-offs: ✅ Strongest "mission-critical" claim; ✅ all README claims survive scrutiny. ❌ Reusable-workflow refactor is the longest single task; ❌ four scanner alert streams demand triage discipline; ❌ probably overruns the 5–6 day P0+P1 budget unless OSS-Fuzz and L3 reusable workflow slip to post-launch.
- Best fit: only if the maintainer is willing to delay public toggle until everything lands, and accepts ongoing alert-triage labor.

**Option B — "Hardened MVP" at launch, L3 + fuzz post-launch (recommended fit for stated 5–6 day budget)**
- Shape: SLSA **L2** via `attest-build-provenance` alone; Trusted Publishing + cosign keyless from day one; CodeQL + OSV-Scanner + dependency-review + Scorecard + harden-runner (audit) at launch; Semgrep added in week 2; cargo-vet imports from Mozilla + Embark only at launch (smallest delta); OSS-Fuzz deferred to post-1.0 with in-tree `cargo-fuzz` harnesses landed as a stretch.
- Trade-offs: ✅ realistic for one solo maintainer in budget; ✅ launch claim is "SLSA L2 with documented L3 roadmap" — defensible. ❌ "SLSA L3" can't appear in launch README; ❌ requires explicit "current state" section in README that survives a year of drift.
- Best fit: Solo maintainer; time-boxed launch; willingness to publish a roadmap that promises L3 and OSS-Fuzz on a public timeline.

**Option C — Conservative launch (no Trusted Publishing, no L3, no OSS-Fuzz)**
- Shape: long-lived `CRATES_IO_TOKEN` in a GitHub Environment with required-reviewer gate; SLSA L2 with attest-build-provenance; Scorecard + dependency-review + CodeQL + harden-runner (audit). Defer Trusted Publishing + cosign + cargo-vet imports + fuzz to a "1.1" release.
- Trade-offs: ✅ minimal moving parts at launch; ✅ smallest blast radius if anything breaks. ❌ undercuts the "mission-critical" framing — long-lived publish tokens are exactly what TP exists to eliminate; ❌ no signed releases on day one is hard to justify in the README.
- Best fit: only if some external constraint (e.g., Trusted Publishing was in fact not GA, which our research shows it *is*) forced the conservative path. Given current TP GA status, this option is dominated by Option B and should be rejected.
