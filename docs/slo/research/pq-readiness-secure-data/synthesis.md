# Synthesis — Post-Quantum Readiness in `secure_data`

## (a) Primary ML-KEM crate to depend on

The choice is between RustCrypto `ml-kem` (pure Rust, unaudited, MIT/Apache, no FIPS path) and `aws-lc-rs` (FFI to AWS-LC, on the FIPS 140-3 modules-in-process list, ring-API-compatible). Project Eleven's 2025 ecosystem survey rates `aws-lc-rs` and `libcrux-ml-kem` as the only "production-ready" options, but `libcrux` is pre-1.0 (`<0.1`) and its maintainers explicitly ask consumers to "talk to us" before production use [12][11]. RustCrypto `ml-kem` v0.3.0 is FIPS-203-final-conformant, KAT-tested, but its README says "USE AT YOUR OWN RISK" [9]. The recommendation is to make the **default `pq` feature back-end RustCrypto `ml-kem`** (pure-Rust, zero new C build deps, fits `secure_data`'s existing dependency posture) and **expose an alternate `pq-aws-lc` feature** that switches to `aws-lc-rs` for any consumer that needs the FIPS 140-3 path. The design must handle backend swappability via a sealed trait inside the `pq` feature gate because no Rust ML-KEM crate today has both an audit and a non-FFI build profile, per Project Eleven's survey [12] and AWS-LC FIPS 3.0's still-pending CMVP status [2].

## (b) Hybrid wire-format choice

Three concrete shapes are in flight: (i) `draft-ietf-tls-hybrid-design-16` (concat: `ML-KEM_ct || X25519_share`, two shared secrets concatenated into the KDF) which is the rustls / AWS / Cloudflare deployed shape [17][7][1][3]; (ii) `draft-irtf-cfrg-hybrid-kems-10` generic QSF combiners which provide stronger IND-CCA2 preservation under adversarial inputs [18]; and (iii) X-Wing (`draft-connolly-cfrg-xwing-kem-10`, 2026-03-02), which fixes the construction at X25519 + ML-KEM-768 + SHA3-256 and produces a 32-byte combined secret — but is still an individual submission with "no formal standing in the IETF standards process" [19]. The pragmatic choice is the **TLS-style concatenation construction** for the wire (matching what AWS KMS, rustls, and Cloudflare have already deployed) but **wrap the combined inputs through an HKDF-SHA-256 KDF before AEAD-key derivation**, which structurally aligns with `draft-irtf-cfrg-hybrid-kems` QSF combiners and gives forward-compatibility room. The envelope must carry a 1-byte combiner-id field so a future flip to X-Wing or to a CFRG-finalised combiner is non-breaking. The design must handle a forward-compatible combiner-id byte in `EncryptionEnvelope` v2 because the IETF/CFRG combiner spec is still a -10 draft and X-Wing is not WG-adopted [19][18].

## (c) FIPS-track recommendation for M4

No FIPS 140-3 certificate covering ML-KEM has been issued as of 2026-05; AWS-LC FIPS 3.0 is the only mainstream module on the CMVP modules-in-process list, since 2024-12-10, with no public ETA [2][16]. NSA CNSA 2.0 mandates ML-KEM for new NSS acquisitions starting 2027-01-01 and full NSS transition by 2035, and EU CRA-adjacent guidance starts critical-infrastructure PQ migration end-of-2026 with 2030 completion [22][25]. Recommendation: **monitor, do not pursue**. M4 should be a written readiness note that (1) names AWS-LC FIPS 3.0 as the candidate validated backend, (2) documents the `pq` + `fips` feature combination as "compiles with `aws-lc-rs`, validation pending CMVP", and (3) commits to upgrading the README claim to "FIPS-track validated" only when a CMVP certificate number exists for the ML-KEM operation. The design must handle a `pq` + `fips` build that is honestly labelled "validation pending CMVP" rather than "FIPS validated" because no module has shipped a 140-3 certificate covering ML-KEM as of 2026-05 [16][2].

## Threat-model framing

The 2026 Mosca/Piani Quantum Threat Timeline Report puts the 10-year probability of a cryptographically relevant quantum computer at 28–49%, the highest in the report's seven-year history [23]. Combined with NSA CNSA 2.0's 2027 acquisition deadline [22] and the EU's 2026/2030 transition timeline [25][26], the migration-plan doc has defensible auditor-facing language: "harvest now, decrypt later" is no longer a hypothetical — major cloud KMS endpoints (AWS, GCP) and >60% of human Cloudflare TLS traffic are already running hybrid PQ in production [1][3][4]. The design must handle ciphertexts produced today remaining decryptable on a 10+ year horizon despite a non-zero probability of CRQC emergence in that window because the Mosca/Piani 2026 report quantifies the risk at 28–49% within 10 years [23].

## KMS integration framing

AWS KMS, GCP Cloud KMS, and HashiCorp Vault Transit have *not* yet shipped above-TLS hybrid-PQ data-key wrapping: AWS KMS's ML-KEM is purely TLS-transport-layer [29][1], GCP exposes ML-KEM as a primitive but does not auto-wrap classical data keys hybrid-style [4], and Vault Transit ML-KEM is on the roadmap but explicitly experimental [6]. This means application-layer hybrid wrapping — exactly the wedge `secure_data`'s `pq` feature occupies — is the only available defence for "harvest now, decrypt later" against ciphertexts produced today, regardless of which KMS backs the `KeyProvider`. The design must handle PQ wrapping at the envelope layer above the existing `KeyProvider` trait without requiring providers to implement PQ themselves because no major KMS exposes hybrid PQ data-key wrapping APIs as of 2026-05 [1][4][6].
