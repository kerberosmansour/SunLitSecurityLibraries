# Synthesis — Formal Verification for SunLit (Kani + TLA+)

## Priority 1 — Kani coverage of the four proposed proofs

The four proofs proposed in the idea doc split cleanly along Kani's known capability boundary. **Deny-by-default** for `secure_authz` and **public-body-no-leak** for `secure_errors::http::into_response_parts` are the simplest: both reduce to "for all bounded inputs, the output satisfies P", with no async, FFI, or unsafe involved — they map onto Kani's sweet spot and are directly comparable to s2n-quic's RTT-estimator and packet-number-decode harnesses, which already run as production CI proofs in 3–45 seconds [source 13]. **Depth/size/field limits** for `secure_boundary` are achievable but require harness discipline: serde-derive expansion can blow up the symbolic state space, so the proof must enter at a hand-written narrow function (e.g., `fn check_depth(input: &Value) -> Result<(), TooDeep>`) rather than ask Kani to symbolically execute the full axum middleware stack — Tokio Bytes set the precedent for this with ghost-state representation invariants on `BytesMut::with_capacity`/`split_off` [source 16]. **Nonce non-zero/unique** on `secure_data` is the one with a real risk: if the AEAD backend dispatches into `aes-ni` or C, Kani cannot follow because FFI and inline assembly are explicitly unsupported [source 1] — the design must handle the AEAD primitive as an opaque axiomatised function (proving only that `secure_data`'s wrapper constructs a non-zero nonce and never reuses one within an envelope, not that the underlying AEAD is correct) because [source 1] states FFI and `asm!` are out of scope for Kani as of 0.67.0 and [source 18] confirms AWS itself splits responsibilities this way: Kani for Rust-level wrapper logic in s2n-quic, separate Coq+Cryptol proofs for aws-lc primitives.

Verdict: **deny-by-default = green, public-body-no-leak = green, depth/size/field limits = green-with-harness-discipline, nonce construction = yellow (achievable with axiomatised AEAD only).** The design must handle the AEAD-as-axiom abstraction explicitly in M3 because [source 18] confirms the AWS pattern is "Kani for the Rust glue, separate proof system for the primitive".

## Priority 2 — TLA+ tool choice and abstraction guidance

`/slo-tla` makes TLC the default and reserves Apalache for state explosion, and the research confirms this is correct for SunLit. The circuit-breaker spec at the bound the idea doc envisions — three states, ≤4 callers, abstract clock-as-tick-sequence, Boolean success/failure — sits comfortably inside TLC's explicit-state range; the closest published-spec analogues are leader-election (Chang-Roberts/Yo-Yo) and termination-detection (EWD840/EWD998) in the TLA+ Examples repo, all of which TLC handles in seconds-to-minutes at comparable bounds [source 26]. The session/step-up spec is structurally similar — a small state set (authenticated, step-up-required, privileged-allowed, expired), bounded actors, abstract clock — and should also fit TLC. The design must handle the abstraction-balance discipline explicitly (drop timestamp granularity, drop user identity to a single role, *preserve* the authenticated-but-not-stepped-up window) because [source 29] shows the only in-the-wild attempt at a circuit-breaker TLA+ spec (nearai/ironclaw, in progress 2026) names exactly this kind of abstraction with explicit invariants like "no Closed→HalfOpen direct transition" and "Open eventually reaches HalfOpen" — reuse the ironclaw invariant list as the M4 starting skeleton; do not wait for the spec text because the issue is open and unmerged. Apalache is the escape hatch only if TLC blows up: the design must handle Apalache's funding/maintenance posture as a known risk because [source 23] confirms it is currently maintained by volunteer effort (Konnov / Kukovec / Pani) after Informal Systems' grant ended in 2024, even though monthly releases continue through April 2026 [source 21]. For OAuth/OIDC-grade protocol verification, Tamarin/ProVerif are the standard, not TLA+ [source 30] — SunLit's `secure_identity` spec models *local* state, not the OIDC handshake, and that scope choice keeps the spec inside TLC's range.

## Priority 3 — CI runtime budget recommendation

The dominant pattern for production Kani lanes is `model-checking/kani-github-action` on every PR, with per-harness times in the seconds-to-tens-of-seconds range; s2n-quic's 30+ harness lane reports per-harness times of 2.8s (packet-number decode) to 44.7s (RTT estimator) [source 13]. Total wall-clock for the full lane is not published, which is itself a signal that a conservative explicit cap is the right policy. The design must handle a **15-minute soft cap on the Kani CI lane (advisory mode for the first release cycle, then promote to blocking)** because [source 8] gives the Kani team's own escape hatch ("if Kani takes too long for your CI, you can try running Kani on a schedule") and confirms GitHub's 6-hour hard kill — 15 minutes leaves comfortable headroom for the four M1–M3 proofs plus harness-startup overhead, with the explicit policy that any new harness pushing the lane past 15 minutes either gets moved to a nightly cron or refactored down to a smaller bound. For the **TLA+ lane, a 10-minute soft cap on TLC is appropriate** at the bounds described in `/slo-tla` (the suitability gate where a naive/broken spec falls over in <1000 reachable states and <10 seconds means a *correct* spec at the same bound is comfortably <10 minutes); the design must handle TLA+ as advisory-only through M4–M5 and not promote to blocking until at least one release cycle of green runs because [source 27] shows even AWS — using TLA+ in production since 2011 — runs TLA+ off-band as a design-time activity rather than per-PR, and SunLit promoting too aggressively risks blocking maintainer velocity on what is fundamentally a design-quality signal. JDK 11+ in the CI image is the only infrastructure prerequisite for TLC [sources 20, 25], and pinning `kani-version` in the workflow YAML is the standard mitigation against nightly-Rust drift breaking Kani lanes [source 3].
